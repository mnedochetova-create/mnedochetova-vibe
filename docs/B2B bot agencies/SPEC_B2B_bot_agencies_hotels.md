# SPEC — B2B‑бот для агентств и отелей (VoC + отзывы)

Связанный PRD: `PRD_B2B_bot_agencies_hotels.md`

## 1. Цели реализации

- обработать выгрузки переписок (WhatsApp/Telegram/CRM/корпоративные чаты) и получить агрегированный отчёт;
- не хранить “сырой” текст переписок после построения отчёта (по умолчанию);
- хранить агрегаты по месяцам для динамики (3 месяца free, 12+ платно);
- собрать репутационные метрики по Яндекс/2ГИС/Google (насколько возможно официально).

## 2. Рекомендуемый стек (быстрый MVP)

- **Backend:** Node.js + TypeScript
- **Telegram bot:** Telegraf
- **Web upload:** простая страница (Express + HTML) или Next.js (если хотим UI)
- **Хранилище:** SQLite (локально) → Postgres (позже)
- **Парсинг файлов:** `xlsx`, `csv-parse`, обработка `.txt/.docx`
- **LLM:** провайдер через API (абстракция `LLMClient`)

## 3. Потоки данных

### 3.1 Импорт

Входы:

- WhatsApp: `.txt`
- Telegram: экспорт `JSON`
- CRM (amoCRM/Bitrix24/TravelLine/самописные) и корпоративные чаты (напр. Webim): `.csv/.xlsx` по схеме

Примечание по Webim:

- У Webim есть API для выгрузки истории диалогов (зависит от тарифа/доступа).
- В MVP **не делаем прямую интеграцию**: принимаем выгрузки агентства в `.csv/.xlsx`. Коннектор к API — отдельным этапом.

Нормализация в единую структуру:

- `thread_id`
- `created_at`
- `channel`
- `agent_name`
- `customer_id`
- `sender` (`agent|customer`)
- `message_text`
- опционально: `product_type`

### 3.2 Обработка (pipeline)

1. validate → 2. normalize → 3. anonymize → 4. topic clustering → 5. root cause extraction → 6. template generation → 7. store aggregates → 8. delete raw

Анонимизация (минимум):

- телефоны → маска
- e-mail → маска
- номера заказов → хэш/маска
- имена (опционально) → маска

### 3.3 Динамика

Храним только агрегаты:

- месяц
- топики/проблемы + частоты
- “новые темы” относительно прошлого месяца
- базовые метрики тональности (если применимо)

## 4. Репутационные метрики

### Google

- официально: Google Places API (Place Details) может возвращать `rating`, `user_ratings_total` и `reviews` (по полям).

### 2ГИС

- официальные API: через Catalog/Search API можно получить карточку организации и метрики (рейтинг/кол‑во оценок — зависит от тарифов/доступа).

### Яндекс

- официальное управление отзывами доступно в Яндекс Бизнес для владельцев карточки.
- сбор отзывов программно может требовать отдельного доступа/инструментов; MVP‑план: поддержать ввод “URL карточки + идентификатор”, а дальше — выбрать легальный способ получения метрик (API/экспорт из кабинета/ручной ввод) в зависимости от доступности.

## 5. Telegram‑интерфейс (MVP)

Команды:

- `/start` — onboarding
- `/upload` — ссылка на веб‑форму загрузки
- `/report` — последний отчёт (pdf/ссылка)
- `/templates` — поиск шаблона (категории + быстрый поиск)
- `/settings` — подключение площадок отзывов (ссылки на карточки)

## 6. Модель данных (SQLite)

Таблицы:

- `orgs` (организации/агентства)
- `imports` (метаданные загрузок: период, количество сообщений, источники)
- `monthly_insights` (агрегаты по месяцам)
- `templates` (шаблоны, версии, категории, product_type, issue_type)
- `review_sources` (площадки и place ids/urls)
- `review_snapshots` (рейтинг/кол-во/топ отзывов по дате)

## 7. Безопасность

- “сырой” файл хранится временно (в памяти/временная папка) и удаляется после обработки
- логирование без PII
- ограничение размера файлов
- rate limit на загрузку/обработку
