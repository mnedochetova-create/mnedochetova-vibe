# SPEC — Семейный travel‑бот‑ассистент в Telegram

Связанный PRD: `PRD_family_travel_bot_assistant_telegram.md`

## 1. Архитектура (MVP)

- Telegram Bot (личка + группы)
- Сервис рекомендаций (LLM)
- Сервис мониторинга (jobs + уведомления)
- Сервис партнёрских ссылок (Travelpayouts)
- Хранилище (SQLite → Postgres позже)

## 2. Основные сущности

- `users`
- `chats` (личка/группа, режим поведения)
- `preferences` (бриф путешествия)
- `tickets` (hotel/flight/tour/poi)
- `watchlists` (мониторинг цен)
- `events` (клики, уведомления)
- `entitlements` (лимиты сообщений/пакеты/подписка)

## 3. Лимит сообщений и монетизация

- считаем **сообщения пользователя** (per user per period)
- два режима оплаты:
  - пакет: +50 сообщений за 199 ₽
  - подписка: 299 ₽/мес (расширенный лимит)
- A/B: разделение пользователей по варианту монетизации (флаг на user)

## 4. “Тикеты” (карточки)

Типы:

- `hotel`
- `flight`
- `tour` (экскурсия)
- `poi` (достопримечательность)

Минимальные поля:

- `type`
- `title`
- `summary`
- `price` (optional)
- `source_url`
- `booking_url` (partner deeplink)
- `meta` (json: город, даты, провайдер, рейтинг и т. д.)

## 5. Мониторинг цен

### 5.1 Ввод пользователя

- flight: `from`, `to`, `dates` (или гибко), `passengers` (опционально)
- hotel: `city`/`hotel_name`, `dates`, `provider` (опционально)

### 5.2 Job модель

- расписание проверки: 1–2 раза в день (на старте)
- хранить историю цен агрегировано (по watchlist)
- триггеры уведомлений: изменение > X% или “новый минимум” (эвристики)

## 6. Интеграции

### Travelpayouts

- генерация deeplink’ов под авиабилеты/отели/экскурсии (в рамках доступных офферов)
- логирование клика и связка с watchlist/ticket

### Places/POI

- MVP: можно начать с “LLM + источники ссылками” (без тяжёлой картографической интеграции)
- позже: подключение картографического провайдера

## 7. Поведение в группах

Настройка `group_mode`:

- `mention_only` (по умолчанию): отвечает только если упомянули
- `on_command`: отвечает на команды (`/plan`, `/watch`, `/tickets`)

Выбор режима — пользователем/админом группы в `/settings`.

## 8. Платежи (Telegram Payments)

- сущности: `invoices`, `payments`, `refunds` (минимум — `payments`)
- выдача entitlement после успешной оплаты

## 9. Безопасность и ограничения

- не хранить лишние персональные данные
- rate limit на команды/мониторинг
- graceful degradation: если внешние сервисы недоступны — бот предлагает альтернативу

## 10. Управление продуктом, аналитика и эксперименты

См. также продуктовый тренд‑разбор: `PRODUCT_TREND_RESEARCH.md`.

**События (event taxonomy)** — для воронки и экспериментов:

- `trip_created`, `ticket_shown`, `partner_click`, `watchlist_created`, `price_alert_sent`, `paywall_shown`, `purchase_completed`
- атрибуты: `source` (dm|group), `ticket_type`, `experiment_id`, `variant`

**Эксперименты**

- флаг пользователя `pricing_variant` (пакет 199 ₽ vs подписка 299 ₽ и др.)
- для A/B — фиксировать вариант при первом визите, не менять в середине сессии без правил

**Качество и guardrails LLM**

- шаблоны ответов с обязательными полями тикета; отказ генерировать юридические гарантии цены
- логирование без PII; при ошибке партнёрского URL — fallback «обновите запрос»

**Расширение модели данных (backlog)**

- `trips` — привязка `preferences`, `tickets`, `watchlists` к одной поездке
- `alert_rules` — порог %, cron, quiet hours
- `experiments` / `user_experiment_assignments` — при росте числа тестов
