# Travelpayouts / Aviasales Data API — что реально приходит в ответах

Цель: зафиксировать **какие поля** можно показывать в карточке перелёта и что **нужно добирать дисклеймером**, без ожиданий «тарифа как на сайте авиакомпании».

Источник описания полей: [Data Access API (readthedocs)](https://travelpayouts-data-api.readthedocs.io/en/latest/), актуальные пути эндпоинтов уточнять в [официальной справке Travelpayouts](https://support.travelpayouts.com/) / кабинете разработчика.

## Безопасность

- Токен Data API передаётся в **`X-Access-Token`** или параметром **`token`**.  
- **Не коммитить** токен в репозиторий; только `TRAVELPAYOUTS_TOKEN` в `.env` (локально) и плейсхолдер в `bot/.env.example`.  
- Если токен светился в чате или в тикете — **сгенерировать новый** в кабинете Travelpayouts.

## Важное ограничение по смыслу «тарифа»

Data API отдаёт **кэшированные цены и агрегаты поиска**, а не полную расшифровку условий тарифа (ручная кладь, багаж, возврат, обмен). Эти условия определяются **конкретным предложением на сайте партнёра после перехода** по партнёрской ссылке.

**В MVP карточки:** показываем то, что стабильно есть в API + короткий текст: *«Багаж, возврат и обмен — на сайте партнёра при бронировании»*.

## Типичные поля по группам эндпоинтов (авиа)

### A) `v2/prices/latest` (и близкие агрегаты «последние цены»)

По документации в ответе встречаются, в частности:

| Поле (API) | Смысл для карточки |
|------------|-------------------|
| `value` | Цена в выбранной валюте |
| `origin`, `destination` | Пункты (IATA города/аэропорты — как вернул API) |
| `depart_date`, `return_date` | Даты вылета/возврата |
| `number_of_changes` | Число пересадок (0 = прямой или условно «без пересадок» в данных) |
| `trip_class` | Класс: 0 эконом, 1 бизнес, 2 первый |
| `found_at` | Когда найдена цена в выдаче |
| `actual` | Флаг актуальности предложения (как в ответе) |
| `distance` | Расстояние (если нужно для контекста) |
| `gate` | В некоторых ответах — продавец/шлюз (если поле есть) |

Часто **нет** отдельных полей: багаж кг, правила возврата, сборы — их **не выводим как факт из API**.

### B) `v1/prices/cheap` / `v1/prices/direct` (дешёвые варианты по направлению)

| Поле (API) | Смысл для карточки |
|------------|-------------------|
| `price` | Цена |
| `airline` | IATA авиакомпании |
| `flight_number` | Номер рейса |
| `departure_at`, `return_at` | Время/даты |
| `expires_at` | До какого момента цена считается актуальной в кэше |

### C) Эндпоинты `v3` (например, календарь / даты — в кабинете партнёра)

Состав полей может отличаться; после выбора конкретных методов для MVP зафиксировать **пример реального JSON** ответа (один запрос с тестовыми параметрами) и обновить эту таблицу.

## Минимальный набор для UI карточки перелёта (MVP)

**Обязательно (если есть в выбранном эндпоинте):**

1. **Цена** (`value` или `price`) + **валюта** (как в запросе/ответе).  
2. **Маршрут** — origin → destination (подписи городов — через справочник IATA у себя в боте).  
3. **Даты** — вылет / возврат (если перелёт туда‑обратно).  
4. **Пересадки** — `number_of_changes` или эквивалент в v3.  
5. **Класс** — из `trip_class`, если запрашивали/получили.  
6. **Свежесть цены** — `found_at` и/или `expires_at` (если есть) — короткой строкой: «цена из кэша на …».  
7. **Дисклеймер** — финальная цена и условия тарифа на стороне продавца после перехода.

**Опционально P1:** авиакомпания / номер рейса (если эндпоинт `cheap`/`direct`), `gate` для прозрачности «где покупать».

**Не обещать из Data API как факт:** багаж, возврат, ужины, выбор места — только «смотрите при бронировании».

## Партнёрка: бронирования, выплаты, сверка с ботом

Бот **не видит** факт оплаты на стороне OTA напрямую; **источник правды по комиссиям и бронированиям** — **личный кабинет Travelpayouts** (отчёты по бронированиям и выплатам за период).

**Практика:**

- Регулярно сверять **отчёты кабинета** с внутренней аналитикой бота (клики по партнёрским ссылкам, `partner_click`, UTM/маркер партнёрской программы — по тому, как настроите ссылки).
- Правила атрибуции и задержки подтверждения брони смотреть в **актуальной справке Travelpayouts** для партнёров (могут отличаться по программе и продукту).
- Цель процесса: **согласовать воронку в боте** с **фактическими начислениями** в кабинете по месяцам/кампаниям.

## Связка с разработкой

1. Положить токен в `bot/.env`: `TRAVELPAYOUTS_TOKEN=...` (файл в `.gitignore`).  
2. Вызовы — с заголовком `Accept-Encoding: gzip, deflate` (рекомендация API).  
3. Зафиксировать в коде **один–два** основных метода для MVP (например, latest + cheap или v3 prices_for_dates — по доступу в вашем кабинете) и парсить поля из реального ответа.

После первого успешного ответа от вашего токена имеет смысл добавить в этот файл **пример JSON** (без токена) для выбранного эндпоинта.
