# MVP Bot Spec — Family Travel Bot

## 1. Цель и границы текущего MVP

### Цель
- Сократить путь группы от хаотичного обсуждения к согласованному и подтверждённому **единому брифу**.

### Что входит в текущий MVP (реализовано в коде)
- Сессия обсуждения поездки в Telegram в DM-потоке (организатор + участники по deep-link).
- Сбор и поэтапное уточнение структурированного **брифа**.
- Подключение участников по ссылке после первичного брифа организатора.
- Подтверждение вклада участником и уведомление организатора.
- Возврат к ранее созданным сессиям через `Мои события`.

### Что не входит в текущий MVP
- Генерация 2-3 направлений и фиксация short-list.
- Голосование, кворум, алгоритм финального выбора.
- Дедлайны, автонапоминания и автозавершение этапов.
- Полноценная групповая оркестрация шагов в чате группы.
- Бронирование, партнерские ссылки, post-purchase сценарии.
- Внешняя верификация визовых/правовых ограничений.

## 2. Роли и каналы

### Роли
- **Организатор**: создает событие, заполняет основной бриф, приглашает участников.
- **Участник**: входит по ссылке, дает персональные вводные, подтверждает корректность своего вклада.
- **Бот**: оркестратор сбора брифа, валидации, обогащения сводки и уведомлений.

### Каналы
- **Личка**: основной рабочий канал текущего MVP (ввод данных, уточнение, подтверждение).
- **Группа**: пока не используется как активный канал сценария (кроме ручной пересылки ссылки организатором).

## 3. State Machine текущего MVP

`start -> event_created -> organizer_brief_collection -> organizer_brief_clarify -> invite_shared -> participant_joined -> participant_contribute -> participant_confirmed -> organizer_notified`

### Правила переходов
- Переход к `invite_shared` происходит только после достаточного первичного брифа организатора.
- Переход к `participant_confirmed` происходит только через явное подтверждение участника кнопкой.
- После `participant_confirmed` бот всегда отправляет уведомление организатору с обновлённой сводкой.
- Возврат в активную сессию возможен из меню `Мои события` для обеих ролей.

## 4. Функциональные требования

### 4.0 Актуализация текущего релиза (бот-прототип)
- В текущем релизе стартовый вход идет через приветствие и кнопку **«Создать событие»** (без ручного выбора роли на первом экране).
- Организатор после создания события сначала заполняет вводные; ссылка для участников показывается после формирования базового брифа.
- Черновик брифа показывается структурированно: пожелания/ограничения участников + факты по поездке.
- Блок "Разногласия" в явном виде не выводится отдельным разделом; сигнал конфликта включается в ограничения.
- Меню в чате ориентировано на текстовые кнопки с иконками: создать событие, что умеет бот, помощь.
- Для форматирования сообщений используется HTML parse mode (жирный, списки, моноширинный текст для дат/значений).
- Состояние сессий сохраняется между перезапусками в файловом хранилище прототипа (`bot/data/events.json`).
- Участник после входа видит текущую сводку события, добавляет вклад, подтверждает корректность и только после этого бот уведомляет организатора.
- Вклад участника обогащает бриф без перезаписи базовых полей организатора (например, бюджета события).
- Раздел `Помощь` работает как центр самовосстановления сценария: показывает контекст (роль/событие/этап), причины уточнений и быстрые действия.
- В главном меню добавлен раздел `Мои события`: бот хранит роли, участников и позволяет вернуться в любую начатую сессию.
- В `Мои события` добавлены статус и подсказка следующего действия для каждого события (role-aware).
- Этап генерации направлений и фиксации short-list пока не реализован в коде и остается следующим этапом.

### 4.1 Сессия и участники
- Организатор создает сессию в личке бота.
- Бот формирует deep-link участника, но показывает его только после первичного заполнения брифа.
- Организатор вручную отправляет ссылку участникам.
- Повторный вход через `/start` и/или `Мои события` возвращает пользователя в актуальный сценарий.
- Для каждого события сохраняются `event_number`, `created_at`, `updated_at`, роль пользователя и прогресс участника.

### 4.2 Сбор данных (бриф)
- Организатор: обязательные 8 полей + обязательное поле "Описание ситуации/пожеланий своими словами".
- Участник: сокращенный набор 4-5 полей + опциональное свободное описание.
- Бот проверяет полноту и формат, показывает недостающие поля.
- Для участника действует отдельный шаг подтверждения: `Подтвердить бриф` / `Дополнить ещё`.
- Семантический парсинг участника отдельно фиксирует дополнительные пожелания (песчаный пляж, поездки к достопримечательностям, рестораны и локальная еда).
- Парсер учитывает длительность поездки (например, 5-7 дней), а также гибкую логику перелета: лимит по часам или допустимость пересадок.
- Вклад участника не перезаписывает уже заполненные базовые поля организатора (например, бюджет события).
- При потере FSM-состояния бот восстанавливает сценарий по данным события из хранилища.

### 4.3 Что зарезервировано на следующий этап
- Генерация рекомендаций (2-3 направления с объяснением).
- Публикация и фиксация short-list.
- Сценарий несогласия, пересборки и ограничение числа циклов.
- Дедлайны, напоминания, автозавершение этапов.

## 5. Требования к данным

Правила качества парсинга и анти-додумывания: `PARSING_SPEC.md`.

### 5.1 Обязательные поля организатора (8)
- Бюджет.
- Состав группы.
- Наличие/возраст детей.
- Даты или гибкость дат.
- Комфортная длительность перелета.
- Климатические предпочтения.
- Визовые/документные ограничения (по словам пользователей).
- Тип отдыха/приоритеты.
- Дополнительно (если указано): длительность поездки (например, 5-7 дней) и формат перелета (допустимы ли пересадки).

### 5.2 Дополнительное текстовое поле
- Обязательно для организатора: контекст, нюансы, ограничения своими словами.
- Опционально для участника: личные пожелания в свободной форме.

### 5.3 Короткий бриф участника (4-5)
- Бюджетный диапазон.
- Даты/гибкость.
- Перелет/длительность.
- Ограничения по детям/здоровью/документам.
- (Опционально) климат/формат отдыха.
- (Опционально) желаемая длительность поездки и предпочтения по пересадкам.

## 6. Экраны (типы сообщений/состояний текущего MVP)

### Организатор — Личка
- Приветствие и старт.
- Создание события.
- Первичный ввод контекста.
- Дозаполнение и редактирование брифа.
- Вывод ссылки для приглашения участников.
- Уведомления о подтверждённых вкладах участников.

### Участник — Личка
- Подключение по deep-link.
- Показ текущей сводки события.
- Ввод сокращенного брифа.
- Показ обновленной сводки с вкладом участника.
- Подтверждение ответов или дополнительное уточнение.

### Системные
- Ошибка LLM/некорректный ответ (парсинг автоматически уходит в rule-based fallback).
- Уведомление организатора о подтвержденном вкладе участника.
- Help-центр с функциями: продолжить сценарий, объяснение белых пятен, помощь по ссылке, переход в `Мои события`, шаблон сообщения о проблеме.

## 6.1 Структура раздела `Мои события` (v1)

- Карточка события: `#номер`, роль пользователя, статус события, следующее действие.
- Статусы v1: `активно`, `нужны уточнения`, `ждем участников`, `завершено`, `архив`.
- Сортировка: сначала события с приоритетным действием, затем по `updated_at`.

## 7. Риски MVP и обязательные контрмеры

- **Риск качества LLM**: строгая схема ответа + пост-валидация + запасной сценарий при низкой уверенности.
- **Расхождение ожиданий без голосования**: единое публичное сообщение короткого списка + явный финальный статус этапа.
- **Асинхронность**: обязательные дедлайны + максимум 2 напоминания + автозавершение.
- **Риск нерелевантности**: обязательные фильтры в брифе + короткий дисклеймер в каждом результате.
- **Риск недоверия к черновым рекомендациям**: явный статус `draft_recommendations` + CTA на дозаполнение перед фиксацией.

## 8. Калибровка разрывов сценария (на текущий момент)

### Закрыто в коде
- Неочевидный переход organizer -> participant закрыт deep-link сценарием и отдельным приветствием участника.
- Неявный шаг после вклада участника закрыт явным подтверждением (`Подтвердить бриф` / `Дополнить ещё`).
- Нет прозрачности для организатора закрыто уведомлением о подтверждённом вкладе участника.
- Потеря контекста между входами закрыта меню `Мои события` и восстановлением роли/сессии.

### Остается как разрыв (следующий этап)
- Нет формализованного статуса `inactive/soft-decline` для участника.
- Нет дедлайнов этапов и автонапоминаний по неответившим.
- Нет этапа рекомендаций и short-list в рабочем коде.
- Нет сценария несогласия с short-list (пересборка/ограничение циклов), так как сам этап short-list еще не внедрен.

## 9. UX-ограничения для простоты

- Один линейный путь: `бриф -> рекомендации -> короткий список`.
- На каждом шаге один основной CTA.
- Технические детали валидации не показывать пользователю.
- Дисклеймер по релевантности: 1-2 коротких предложения.

## 10. Поэтапная реализация

### Этап 1. Оркестрация
- Создание сессии, приглашения, статусы участника, дедлайны.

### Этап 2. Brief
- Формы organizer/participant, валидация обязательных полей, fallback-ветка.

### Этап 3. Рекомендации (следующий этап)
- LLM-интеграция, структурированный ответ, публикация 2-3 направлений.

### Этап 4. Завершение этапа (следующий этап)
- Подтверждение короткого списка или автозакрытие по дедлайну.
- Один цикл пересборки по запросу.

### Этап 5. Hardening
- Логирование событий.
- Обработка edge-cases.
- Базовые метрики этапа.

## 11. Acceptance Criteria (Given/When/Then)

- **AC-1 Deep-link переход**
  - Given сессия создана и ссылка отправлена участнику
  - When participant нажимает deep-link
  - Then личка открывает связанную сессию без ручного поиска.

- **AC-2 `/start` fallback**
  - Given participant отправляет `/start` без deep-link
  - When есть активные инвайты
  - Then bot показывает список сессий и продолжает сценарий после выбора.

- **AC-3 Подтверждение участника**
  - Given участник отправил свои вводные
  - When бот показывает обновленный бриф
  - Then участник может подтвердить вклад или отправить уточнение.

- **AC-4 Уведомление организатора**
  - Given участник подтвердил вклад
  - When бот получает подтверждение
  - Then организатор получает сообщение о том, кто дополнил бриф, с актуальной сводкой.

- **AC-5 Готовность к следующему этапу**
  - Given заполнены 8 полей organizer
  - And есть минимум 1 валидный ответ участника в личке
  - And заполнено обязательное свободное описание organizer
  - When запускается проверка готовности
  - Then бот фиксирует готовность брифа и переводит сценарий к следующему этапу разработки (рекомендации).

- **AC-6 Блокировка при недоборе**
  - Given не выполнено хотя бы одно условие готовности
  - When organizer запускает генерацию
  - Then бот показывает недостающие поля и предлагает дозаполнение или fallback.

- **AC-7/8/9/10/11**
  - Зафиксированы как критерии следующего этапа (short-list, дедлайны, пересборка, автозавершение).
  - В текущем релизе не валидируются, так как соответствующие шаги еще не реализованы.

## 12. Post-MVP

- Голосование и кворум.
- Алгоритм финального выбора направления.
- Обработка tie-cases и эскалации.
