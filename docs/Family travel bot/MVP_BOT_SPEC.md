# MVP Bot Spec — Family Travel Bot

## 1. Цель и границы MVP

### Цель
- Сократить путь группы от хаотичного обсуждения к согласованному **короткому списку** из 2-3 направлений.

### Что входит в MVP
- Сессия обсуждения поездки в Telegram (`group + dm`).
- Сбор структурированного **брифа**.
- Генерация 2-3 направлений с объяснением.
- Публикация и фиксация **короткого списка**.

### Что не входит в MVP
- Голосование, кворум, алгоритм финального выбора.
- Бронирование, партнерские ссылки, post-purchase сценарии.
- Внешняя верификация визовых/правовых ограничений.

## 2. Роли и каналы

### Роли
- **Организатор**: запускает сессию, заполняет основной бриф, завершает этап.
- **Участник**: подтверждает участие, дает персональные вводные, может запросить пересборку.
- **Бот**: оркестратор этапов, валидация данных, генерация короткого списка, контроль дедлайнов.

### Каналы
- **Группа**: общий статус сессии, приглашения, публикация результатов.
- **Личка**: персональный ввод данных и подтверждения.

## 3. State Machine MVP

`init -> participant_confirmation -> brief_collection -> brief_validation -> recommendations -> shortlist_published -> (shortlist_confirmed | closed_by_timeout | revision_requested_once | insufficient_group_input)`

### Правила переходов
- `confirmed -> active`: по первому валидному ответу участника в личке.
- `inactive`: только при явном отказе (soft-decline) от участника.
- `revision_requested_once`: разрешен только один цикл пересборки в MVP.
- `insufficient_group_input`: нет ни одного валидного participant-ответа до дедлайна.

## 4. Функциональные требования

### 4.0 Актуализация текущего релиза (бот-прототип)
- В текущем релизе стартовый вход идет через приветствие и кнопку **«Создать событие»** (без ручного выбора роли на первом экране).
- Организатор после создания события сразу получает ссылку для участников.
- Черновик брифа показывается структурированно: пожелания/ограничения участников + факты по поездке.
- Блок "Разногласия" в явном виде не выводится отдельным разделом; сигнал конфликта включается в ограничения.
- Меню в чате ориентировано на текстовые кнопки с иконками: создать событие, что умеет бот, помощь.
- Для форматирования сообщений используется HTML parse mode (жирный, списки, моноширинный текст для дат/значений).
- Хранилище событий и брифов в прототипе временное (`in-memory`) и очищается при перезапуске.
- Участник после входа видит текущую сводку события, добавляет вклад, подтверждает корректность и только после этого бот уведомляет организатора.
- Вклад участника обогащает бриф без перезаписи базовых полей организатора (например, бюджета события).
- В главном меню добавлен раздел `Мои события`: бот хранит роли, участников и позволяет вернуться в любую начатую сессию.
- Состояние сессий сохраняется между перезапусками (файловое хранилище прототипа).
- Этап генерации направлений и фиксации short-list пока не реализован в коде и остается следующей итерацией MVP.

### 4.1 Сессия и участники
- Организатор создает сессию и дедлайны этапов.
- Бот публикует приглашение в группе со ссылкой на вход в личку.
- Запасной путь: `/start` в личке с выбором активной сессии.

### 4.2 Сбор данных (бриф)
- Организатор: обязательные 8 полей + обязательное поле "Описание ситуации/пожеланий своими словами".
- Участник: сокращенный набор 4-5 полей + опциональное свободное описание.
- Бот проверяет полноту и формат, показывает недостающие поля.
- Для участника действует отдельный шаг подтверждения: `Подтвердить бриф` / `Дополнить ещё`.
- Семантический парсинг участника отдельно фиксирует дополнительные пожелания (песчаный пляж, поездки к достопримечательностям, рестораны и локальная еда).

### 4.3 Генерация рекомендаций
- Источник: LLM (генерация по брифу).
- Ответ модели принимается только в строгой структуре.
- Бот делает пост-валидацию и публикует 2-3 направления с объяснениями.

### 4.4 Фиксация короткого списка
- Короткий список публикуется единым сообщением в группе.
- Организатор завершает этап кнопкой подтверждения.
- Если дедлайн завершения истек без подтверждения -> `closed_by_timeout`.

### 4.5 Несогласие и пересборка
- Участники в группе могут нажать "Запросить пересборку".
- Организатор подтверждает пересборку.
- Повторная генерация допускается только один раз.

## 5. Требования к данным

### 5.1 Обязательные поля организатора (8)
- Бюджет.
- Состав группы.
- Наличие/возраст детей.
- Даты или гибкость дат.
- Комфортная длительность перелета.
- Климатические предпочтения.
- Визовые/документные ограничения (по словам пользователей).
- Тип отдыха/приоритеты.

### 5.2 Дополнительное текстовое поле
- Обязательно для организатора: контекст, нюансы, ограничения своими словами.
- Опционально для участника: личные пожелания в свободной форме.

### 5.3 Короткий бриф участника (4-5)
- Бюджетный диапазон.
- Даты/гибкость.
- Перелет/длительность.
- Ограничения по детям/здоровью/документам.
- (Опционально) климат/формат отдыха.

## 6. Экраны (типы сообщений/состояний)

### Организатор — Группа
- Старт сессии.
- Состав участников и статусы.
- Прогресс заполнения брифа.
- Публикация и закрепление короткого списка.

### Организатор — Личка
- Первичный ввод контекста.
- Дозаполнение и редактирование.
- Выбор: дозаполнить или идти в fallback.
- Подтверждение завершения этапа.

### Участник — Группа
- Инвайт в сессию.
- Текущий статус этапа.
- Итоговый короткий список.
- Кнопка "Запросить пересборку" (до дедлайна).

### Участник — Личка
- Подтверждение участия.
- Показ текущей сводки события.
- Ввод сокращенного брифа.
- Показ обновленной сводки с вкладом участника.
- Подтверждение ответов или дополнительное уточнение.

### Системные
- Ошибка LLM/некорректный ответ.
- Статус `draft_recommendations`.
- Напоминания и автозавершение по дедлайну.
- Уведомление организатора о подтвержденном вкладе участника.

## 7. Риски MVP и обязательные контрмеры

- **Риск качества LLM**: строгая схема ответа + пост-валидация + запасной сценарий при низкой уверенности.
- **Расхождение ожиданий без голосования**: единое публичное сообщение короткого списка + явный финальный статус этапа.
- **Асинхронность**: обязательные дедлайны + максимум 2 напоминания + автозавершение.
- **Риск нерелевантности**: обязательные фильтры в брифе + короткий дисклеймер в каждом результате.
- **Риск недоверия к черновым рекомендациям**: явный статус `draft_recommendations` + CTA на дозаполнение перед фиксацией.

## 8. UX-ограничения для простоты

- Один линейный путь: `бриф -> рекомендации -> короткий список`.
- На каждом шаге один основной CTA.
- Технические детали валидации не показывать пользователю.
- Дисклеймер по релевантности: 1-2 коротких предложения.

## 9. Поэтапная реализация

### Этап 1. Оркестрация
- Создание сессии, приглашения, статусы участника, дедлайны.

### Этап 2. Brief
- Формы organizer/participant, валидация обязательных полей, fallback-ветка.

### Этап 3. Рекомендации
- LLM-интеграция, структурированный ответ, публикация 2-3 направлений.

### Этап 4. Завершение этапа
- Подтверждение короткого списка или автозакрытие по дедлайну.
- Один цикл пересборки по запросу.

### Этап 5. Hardening
- Логирование событий.
- Обработка edge-cases.
- Базовые метрики этапа.

## 10. Acceptance Criteria (Given/When/Then)

- **AC-1 Deep-link переход**
  - Given сессия создана и инвайт опубликован в группе
  - When participant нажимает deep-link
  - Then личка открывает связанную сессию без ручного поиска.

- **AC-2 `/start` fallback**
  - Given participant отправляет `/start` без deep-link
  - When есть активные инвайты
  - Then bot показывает список сессий и продолжает сценарий после выбора.

- **AC-3 Неактивность только по soft-decline**
  - Given participant `confirmed` или `active`
  - When нажимает "Не участвую сейчас"
  - Then получает статус `inactive` с фиксацией причины.

- **AC-4 Молчание не меняет статус**
  - Given participant не отвечает до дедлайна
  - When дедлайн истекает
  - Then статус не меняется на `inactive` автоматически.

- **AC-5 Готовность к короткому списку**
  - Given заполнены 8 полей organizer
  - And есть минимум 1 валидный ответ участника в личке
  - And заполнено обязательное свободное описание organizer
  - When запускается проверка готовности
  - Then бот переходит к генерации.

- **AC-6 Блокировка при недоборе**
  - Given не выполнено хотя бы одно условие готовности
  - When organizer запускает генерацию
  - Then бот показывает недостающие поля и предлагает дозаполнение или fallback.

- **AC-7 Завершение по подтверждению**
  - Given короткий список опубликован
  - When organizer подтверждает этап
  - Then сессия закрывается со статусом `shortlist_confirmed`.

- **AC-8 Автозавершение**
  - Given короткий список опубликован, но organizer не подтвердил этап
  - When наступает дедлайн завершения
  - Then сессия закрывается со статусом `closed_by_timeout`.

- **AC-9 Один цикл пересборки**
  - Given есть запрос "Запросить пересборку"
  - When organizer подтверждает пересборку
  - Then запускается один дополнительный цикл доуточнения и генерации.

- **AC-10 Нет второй пересборки**
  - Given одна пересборка уже была
  - When появляется повторный запрос
  - Then бот блокирует повтор и переводит сценарий в post-MVP.

- **AC-11 Нулевой отклик**
  - Given дедлайн сбора истек и нет ни одного валидного participant-ответа
  - When бот завершает этап
  - Then сессия получает статус `insufficient_group_input` и organizer выбирает: закрыть или fallback.

## 11. Post-MVP

- Голосование и кворум.
- Алгоритм финального выбора направления.
- Обработка tie-cases и эскалации.
