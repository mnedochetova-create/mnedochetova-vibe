# Product Card — Family travel bot (MVP-focus: выбор направления)

## 0) Как читать карточку
- Этот файл фиксирует **самый первый этап**: помочь группе **договориться о направлении до поездки**.
- Сценарий "обсуждение -> бронирование + момент покупки", монетизация, метрики и прочие расширения перенесены в `BACKLOG_NEXT_STAGES.md`.
- Детальная спецификация MVP для разработки: `MVP_BOT_SPEC.md`.
- Источники: `Рынок и исследования/Интервью_и_исследования.md`, `USER_FLOWS.md`.

## 1) Узкая целевая аудитория (2 роли)

### Роль 1 — Организатор поездки (primary user)
- Инициирует поездку и координирует обсуждение в Telegram.
- Обычно собирает критерии "в голове/чате", потом вручную сводит мнения участников.
- Боль: долгое согласование и потеря темпа решения.

### Роль 2 — Участники поездки (co-users)
- Члены семьи/друзья, которые влияют на финальный выбор направления.
- У каждого свои ограничения: бюджет, ребенок, удобство перелета, предпочтения по климату.
- Боль: сложно быстро понять общий компромисс и договориться без конфликтов/затяжки.

## 2) Одна проблема, с которой начинаем
- **Проблема MVP:** группа не может быстро договориться о направлении поездки в чате, потому что критерии разбросаны по сообщениям, а участники отвечают асинхронно.
- Следствие: решение откладывается на дни/недели, организатор устает, поездка "зависает".

## 3) Одна killer feature для первой проверки
- **Killer feature MVP:** `Групповой сбор брифа -> 2-3 направления с объяснением "почему это подходит группе"`.
- Что проверяем: бот сокращает время от хаотичного обсуждения до согласованного короткого списка из 2-3 направлений.

## 4) Откуда берем данные для MVP
- **Входные данные от пользователей:** критерии в чате (бюджет, состав, дети, визы, длительность перелета, климат, даты/гибкость).
- **Внутренние данные MVP:** структурированный бриф и ответы участников внутри Telegram-диалога.
- **Внешние API на этом этапе:** не обязательны для первой проверки гипотезы (фокус на согласовании направления).

## 5) Клиентский сценарий / Job
- **Job (core):** "Когда мы обсуждаем поездку в чате и не можем договориться, я хочу, чтобы бот собрал требования всех участников и предложил 2-3 направления с понятной логикой, чтобы мы быстро приняли решение по направлению."

### Сценарий MVP (один сквозной)
1. В личке бот приветствует и предлагает действие: **создать событие**.
2. Организатор отправляет вводные одним сообщением.
3. Бот собирает структурированный черновик брифа (пожелания/ограничения + факты) и дозапрашивает только недостающее.
4. После готовности брифа бот дает ссылку-приглашение для участников.
5. Участники входят по ссылке, получают текущую сводку события и добавляют свои предпочтения одним сообщением.
6. Бот структурирует вклад участника, показывает обновлённый бриф и просит подтверждение участника.
7. После подтверждения бот отправляет организатору уведомление о том, кто дополнил бриф.
8. На текущем этапе рекомендации и фиксация short-list остаются следующим шагом разработки.

## 6) Критерии успеха текущего этапа (MVP)
- Заполнен бриф по ключевым критериям.
- Участники подключаются по ссылке и дополняют бриф.
- Участник подтверждает корректность своего вклада.
- Организатор получает уведомление об обновлении брифа от участника.
